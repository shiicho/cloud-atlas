# 05 · Slack/メール の作法 — 处理模糊请求的艺术

> **目标**：学会优雅地处理模糊请求，展示主动判断力  
> **核心模式**：曖昧な依頼への対応  
> **难度**：高级

---

## 为什么重要 (Why This Matters)

你的上司在 Slack 发来一条消息：

> 「この画面、なんか使いにくいから、いい感じにしといて」
> （这个画面不太好用，帮我弄好点）

你心里想："什么叫'いい感じ'？能不能说清楚点？"

于是你回复：

> 「具体的に何をすればいいですか？要件を明確にしてください。」
> （具体要做什么？请明确需求。）

恭喜，你刚刚：
- 把问题推回给上司
- 暗示他的指示不够专业
- 错失了一次展示能力的机会

**在日本职场，模糊指示不是上司的问题——而是你展示判断力的机会。**

---

## 文化模式解析 (Cultural Pattern)

### 曖昧な依頼への対応

| 日语 | 含义 |
|------|------|
| 曖昧な依頼 (Aimai na Irai) | 模糊的请求 |
| いい感じに | "弄好点"、"让它感觉不错" |
| よしなに | "看着办"、"随你处理" |

### 为什么日本上司会这样说？

这不是他们懒，而是：

1. **高语境文化**：期望你能理解言外之意
2. **授权信号**：给你空间展示判断力
3. **关系测试**：看你是否与他"同频"

### 正确的处理方式

```
错误做法：问"你到底要什么？"
         → 把责任推回去 → 显得被动

正确做法：提出具体方案让他选
         → 展示主动性 → 显得专业
```

**核心原则**：**不要问"你要什么"，而是说"我建议这样做，您看行吗"**

## 实用模板 (Template)

### 模板 1：处理模糊请求的回复

上司说：「この画面、いい感じにしといて」

```
@Tanaka-san

承知いたしました。
XX画面のUI改善について、いくつか案を考えてみました。

【A案】検索フォームを画面上部に固定表示
  - メリット：いつでも検索できる
  - デメリット：表示領域が少し狭くなる

【B案】ドラッグ＆ドロップで項目の並び替え
  - メリット：ユーザーがカスタマイズできる
  - デメリット：実装に2日ほどかかる

個人的には、まずA案から着手するのが
費用対効果が高いかと考えておりますが、いかがでしょうか。

ご意見いただけますと幸いです。
```

**要点**：
- 提出 2-3 个具体方案
- 说明优缺点
- 给出自己的推荐
- 请求确认

---

### 模板 2：Code Review 请求

```
@Tanaka-san @Sato-san

XX機能のPRを作成しましたので、
レビューをお願いいたします。

PR: [链接]

**[変更の概要]**
JIRA-456の対応で、決済処理に新しい支払い方法を追加しました。

**[変更ファイル]**
- `src/payment/PaymentService.ts` (主な変更)
- `src/payment/types.ts` (型定義追加)
- `tests/payment.test.ts` (テスト追加)

**[特に見ていただきたい点]**
- PaymentService.ts のエラーハンドリング部分
  → 複雑になってしまったので、より良い書き方があればご意見ください
- DBトランザクションの範囲が適切かご確認いただきたいです

**[テスト結果]**
- 単体テスト: All passed
- ローカル環境での動作確認済み

お忙しいところ恐れ入りますが、
ご確認のほど、よろしくお願いいたします。
```

**要点**：
- 说明变更概要（让 reviewer 有 context）
- 列出关键变更文件
- **指出希望重点审查的地方**（体贴 reviewer）
- 说明测试状态
- 礼貌结尾

## 对比示例 (Bad → OK → Good)

**场景**：上司说「来週のデモ、派手なやつお願いね！」（下周的 demo，搞个酷炫的！）

### ❌ Bad Example：直接问要什么

```
「派手な」とはどういう意味ですか？
具体的な要件を教えてください。
```

**问题分析**：
- 把问题推回给上司
- 暗示他的指示不清楚
- 显得被动、缺乏主动性

**上司的感受**：「自分で考えろよ...」

---

### 🔶 OK Example：只问一个问题

```
承知しました。
アニメーションを増やす方向でよろしいでしょうか？
```

**分析**：
- 比直接问要好
- 但只提了一个选项
- 上司还是需要自己想

---

### ✅ Good Example：提出具体方案

```
@Watanabe-san

承知いたしました！「派手な」デモですね。
いくつかアイデアを考えてみました。

【A案】リアルタイムデータ可視化
  - ダッシュボードにライブグラフを追加
  - 見た目のインパクト大、実装は1日程度

【B案】ドラッグ＆ドロップUI
  - 管理画面でデータ操作がスムーズに
  - UX改善をアピールできる、実装2日

【C案】ダークモード切り替えアニメーション
  - トグルでパッと切り替わる
  - モダンなUI感、実装半日

個人的にはA案が「おっ」となるインパクトがあると思いますが、
どの方向がイメージに近いでしょうか？
```

**为什么这个版本好？**

| 要素 | 体现 |
|------|------|
| 把模糊变具体 | 3 个可选方案 |
| 展示判断力 | 每个方案都有理由 |
| 给出推荐 | 「個人的にはA案が...」 |
| 请求确认 | 「どの方向がイメージに近いでしょうか」 |

**上司的感受**：「お、ちゃんと考えてくれたな。頼りになる。」

## 角色扮演场景 (5-min Role-play)

### 情景设定

产品经理渡辺さん在 Slack 发消息：

```
来週のデモ、派手なやつお願いね！
```

这是一个内部管理工具的 demo。你完全不知道"派手"是什么意思。

### 你的任务

写一条 Slack 回复：
- **不要**直接问"什么叫派手"
- **要**提出 2-3 个具体想法
- **要**给出你的推荐

### 参考回复

```
@渡辺さん

承知しました！来週のデモですね。

「派手」ということで、いくつかアイデアを考えてみました：

1️⃣ ライブデータ同期デモ
   → 画面を2つ開いて、リアルタイムでデータが反映される様子を見せる

2️⃣ 検索の高速レスポンス
   → 大量データでもサクサク動く様子をライブで見せる

3️⃣ ワンクリックレポート生成
   → ボタン1つでPDFが即座に生成される

個人的には①のライブ同期が一番「おおっ」となるかなと思いますが、
どの方向がイメージに近いでしょうか？

明日までにプロトタイプを用意できますので、
ご確認後すぐ着手します！
```

## 失误恢复 (Recovery Line)

### 场景：你刚才直接问了"要什么"，发现对方皱眉了

### 日语恢复用语

```
申し訳ありません、聞き方がストレートすぎました。

私の方でいくつか案を考えてみます。
例えば〜〜という方向性で考えておりますが、
認識は合っておりますでしょうか。
```

### 解释

| 部分 | 作用 |
|------|------|
| 聞き方がストレートすぎました | 承认自己问得太直接 |
| 私の方でいくつか案を考えてみます | 主动承担思考工作 |
| 例えば〜〜 | 立即提出一个具体方向 |
| 認識は合っておりますでしょうか | 请求确认 |

**要点**：承认 → 主动思考 → 提出方案

## 关东 vs 关西 (Regional Nuance)

| 地区 | 模糊指示的频率 |
|------|----------------|
| **关东** | 稍微会给更具体的指示 |
| **关西** | 「いい感じにしといて」「よしなに」更常见 |

### 关西版模糊请求

```
これ、ええ感じにしといてや。
任せるわ。
```

（这个弄好点。交给你了。）

**核心**：无论地区，**用提案代替提问**的原则是一样的。

## 职业进阶 (Bamboo Ceiling Breaker)

### 从"执行者"到"规格制定者"

| 级别 | 能力 |
|------|------|
| **初级工程师** | 需要详细规格才能开始工作 |
| **高级工程师** | 能把模糊目标变成具体规格 |
| **架构师/主管** | 定义规格让别人执行 |

**处理模糊请求的能力**正是从初级到高级的分水岭。

### 「いい感じに」测试

当上司说「いい感じにしといて」，实际上是在测试：

> 「この人は自分で考えて動けるか？
> 　それとも細かく指示しないとダメか？」
> （这个人能自己思考行动吗？还是需要我手把手教？）

通过这个测试的方法：**展示你的判断力**。

### 晋升的关键

公司提拔高级职位时，考虑的是：

> 「この人に任せたら、自分で考えて進めてくれるか？」
> （把事情交给这个人，他能自己想办法推进吗？）

每次处理模糊请求，都是证明"Yes"的机会。
