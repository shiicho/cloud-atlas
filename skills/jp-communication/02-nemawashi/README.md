# 02 · 根回し（Nemawashi）实战

> **目标**：学会在正式提案前进行非正式共识建立，提高方案通过率  
> **核心模式**：根回し (Nemawashi)  
> **难度**：高级

---

## 为什么重要 (Why This Matters)

你是不是经历过这样的场景？

- 你准备了一个很棒的技术方案，在会议上激情演示
- 演示结束，全场沉默
- 上司说"我们再考虑考虑"，然后方案石沉大海

问题出在哪里？

> **你的方案可能很好，但你跳过了"根回し"。**

在西方文化中，在大会上展示一个惊艳的想法是英雄行为。但在日本：

> 在会议上"突然"提出新想法 = 让别人措手不及 = 失礼

日本的会议**不是用来讨论新想法的**，而是用来**确认已经达成的共识**。

真正的决策发生在会议之前。

---

## 文化模式解析 (Cultural Pattern)

### 根回し (Nemawashi)

| 日语 | 字面意思 | 实际含义 |
|------|----------|----------|
| 根回し | 挖掘根部周围（移植树木前的准备） | 正式提案前的非正式沟通和共识建立 |

### 工作原理

```
         西方模式                         日本模式
    ┌─────────────────┐            ┌─────────────────┐
    │   准备方案       │            │   准备方案       │
    └────────┬────────┘            └────────┬────────┘
             │                              │
             ▼                              ▼
    ┌─────────────────┐            ┌─────────────────┐
    │   会议上提案     │            │  1-on-1 根回し   │ ← 真正的讨论
    │   (热烈讨论)     │            │  (收集反馈)     │
    └────────┬────────┘            └────────┬────────┘
             │                              │
             ▼                              ▼
    ┌─────────────────┐            ┌─────────────────┐
    │   投票/决定      │            │  修改方案       │
    └─────────────────┘            └────────┬────────┘
                                            │
                                            ▼
                                   ┌─────────────────┐
                                   │   正式会议       │ ← 只是确认仪式
                                   │  (确认共识)     │
                                   └─────────────────┘
```

### 为什么需要根回し？

1. **避免让人当众难堪**：在会议上反对别人 = 让对方丢脸
2. **收集真实反馈**：1-on-1 时人们更愿意说实话
3. **建立同盟**：提前说服关键人物
4. **预防阻力**：提前发现反对意见，调整方案

## 实用模板 (Template)

### 技术提案根回し检查清单

**我的提案**：[例：从 Jenkins 迁移到 GitHub Actions]

```
[ ] 1. 识别关键利益相关者
       - 决策者：谁有最终决定权？→ 山本部長
       - 影响者：谁的意见有分量？→ 佐藤 Tech Lead
       - 可能反对者：谁会受到负面影响？→ 鈴木さん (Jenkins 专家)

[ ] 2. 准备根回し谈话要点
       - 问题现状：当前 Jenkins 的痛点是什么？
       - 方案概要：GitHub Actions 能解决什么？
       - 征求意见：您觉得这个方向可行吗？

[ ] 3. 按顺序进行 1-on-1
       ① 先找可能反对的人 → 鈴木さん
          "鈴木さん、Jenkins のことでお知恵を借りたいのですが..."
       ② 再找影响者 → 佐藤 Tech Lead
          "鈴木さんにも相談したのですが..."
       ③ 最后找决策者 → 山本部長
          "佐藤さんと鈴木さんにも確認したところ..."

[ ] 4. 收集反馈，修改方案
       - 谁支持？为什么？
       - 谁反对？什么顾虑？
       - 如何调整方案来解决顾虑？

[ ] 5. 正式会议
       - 已经有同盟支持
       - 反对意见已被预先解决
       - 会议只是"走流程"
```

### 根回し开场白模板

找人聊时的开场白：

```
[Name]さん、お忙しいところすみません。
少しお時間いただけますでしょうか。

実は、[话题]についてアイデアがありまして、
[Name]さんのご意見を伺いたいと思っております。

5分ほどよろしいでしょうか。
```

## 对比示例 (Bad → OK → Good)

**场景**：你想提议将认证模块重构，提升性能

### ❌ Bad Example：无根回し，突然提案

在团队会议上：

```
今のアーキテクチャは非効率です。
認証モジュールを完全に書き直すべきです。
こちらが30ページの提案書です。
```

**问题分析**：
- 突然提出 = 让所有人措手不及
- 批评现状 = 可能得罪负责人
- 30 页 = 没人有时间当场消化
- 所有人被迫当众表态 = 不舒服

**结果**：沉默，然后"我们再考虑考虑"

---

### 🔶 OK Example：部分根回し

你提前跟上司说了：

```
田中さん、リファクタリングについて提案があります。
次の会議でプレゼンさせていただいてもよろしいでしょうか。
```

**分析**：
- 通知了上司 ✓
- 但没有征求其他人意见 ✗
- 没有收集反馈 ✗
- 会议上仍然可能遇到阻力

---

### ✅ Good Example：完整根回し

**第一步**：找可能反对的人（鈴木さん）

```
你：鈴木さん、お忙しいところすみません。
    認証モジュールのパフォーマンスについて、
    少しお知恵を借りたいのですが、お時間いただけますか。

鈴木：うん、いいよ。

你：ありがとうございます。最近、認証のレスポンスが
    遅くなってきていて、少しリファクタリングを
    検討しているのですが...鈴木さんから見て、
    どの部分が一番リスクがありそうですか？

鈴木：そうだな、DBのコネクションプーリングが
    怪しいと思ってたんだよね...

你：なるほど！とても参考になります。
    その点も含めて、一度案をまとめてみます。
    また相談させてください。
```

**第二步**：找影响者（佐藤 Tech Lead）

```
你：佐藤さん、認証モジュールの件で相談があります。
    鈴木さんにも少しお話を聞いたのですが、
    パフォーマンス改善のアイデアがありまして...

佐藤：ほう、どんな感じ？

你：[简要说明方案，包含鈴木さん的反馈]

佐藤：いいんじゃない。ただ、工数がどれくらいかは
    ちゃんと見積もった方がいいね。

你：おっしゃる通りです。見積もりも含めて
    資料にまとめます。
```

**第三步**：找决策者（田中部長）

```
你：田中さん、お時間よろしいでしょうか。
    認証モジュールのリファクタリングについて、
    佐藤さんと鈴木さんにも相談したのですが、
    前向きなフィードバックをいただけました。

    次の定例で正式に提案させていただいても
    よろしいでしょうか。

田中：いいよ。資料を事前に共有してくれる？

你：承知しました。明日中に共有いたします。
```

**正式会议**：

```
你：以前佐藤さんと鈴木さんにも相談させていただいた
    認証モジュールの件ですが、正式にご提案いたします。
    [プレゼン]

田中：うん、いいんじゃないかな。みんなどう？

佐藤：いいと思います。（已经被根回し过）

鈴木：僕も賛成です。（他的意见已被采纳）

田中：じゃ、進めてください。
```

## 角色扮演场景 (5-min Role-play)

### 情景设定

你想提议将 CI/CD 从 Jenkins 迁移到 GitHub Actions。

- **山本部長**：决策者，对技术细节不太了解
- **鈴木さん**：资深工程师，Jenkins 的主要维护者，可能会反对

### 你的任务

练习与鈴木さん的第一次根回し对话。

**目标不是说服他**，而是：
1. 表示尊重（征求他的意见）
2. 了解他的顾虑
3. 让他觉得自己被重视

### 参考对话

```
你：鈴木さん、お忙しいところすみません。
    CI/CDのことで、少しご相談があるのですが、
    5分ほどお時間いただけますか。

鈴木：うん、いいよ。

你：ありがとうございます。実は最近、GitHub Actions を
    調べていて、いくつかメリットがあるかなと
    思っているのですが...

    鈴木さんはJenkinsを長年運用されてきたので、
    もし移行するとしたら、どのあたりが
    リスクになりそうか、ご意見伺えますか？

鈴木：うーん、Jenkinsは確かに古いけど、
    カスタマイズが多いからな。移行は大変だと思うよ。

你：なるほど、やっぱりそうですよね。
    そのあたりのカスタマイズ、具体的にはどんなものが
    ありますか？ちゃんと把握しておきたいので。

鈴木：[具体的に説明]

你：すごく参考になります。そのあたりも含めて、
    一度整理してみます。また相談させてください。

鈴木：ああ、いいよ。
```

**要点**：
- 征求意见，不是说服
- 承认他的专业知识
- 请他指出风险
- 表示会把他的意见纳入考虑

## 失误恢复 (Recovery Line)

### 场景：在会议上提案，遭遇沉默/阻力

你意识到应该先做根回し，现在需要优雅地撤退。

### 日语恢复用语

```
申し訳ございません、
少し説明を急ぎすぎたかもしれません。

皆様のご意見を伺いながら、
改めて検討したいと思いますので、

まずはこの件、一旦持ち帰らせていただけますでしょうか。
```

### 解释

| 部分 | 作用 |
|------|------|
| 急ぎすぎた | 承认自己操之过急 |
| ご意見を伺いながら | 表示会征求意见（= 做根回し） |
| 持ち帰らせていただけますか | 请求收回方案，重新准备 |

**要点**：体面地撤退，然后**私下做根回し**，下次再正式提案。

## 关东 vs 关西 (Regional Nuance)

| 地区 | 根回し风格 |
|------|------------|
| **关东** | 较正式：预约时间、准备资料、书面沟通为主 |
| **关西** | 较随意：可能在午餐、咖啡、居酒屋进行 |

### 关西版根回し开场白

```
今度、飯でも行きまへんか？
ちょっと相談したいことがあって...
```

（下次一起吃个饭？有点事想请教...）

**核心**：无论形式如何，本质都是一样的——**正式提案前的非正式共识建立**。

## 职业进阶 (Bamboo Ceiling Breaker)

### 从"执行者"到"领导者"

中国工程师常见的晋升障碍：

> 「アイデアはいいんだけど、進め方がね...」
> （想法是不错，但推进方式不太对...）

不做根回し的人被认为是：
- 不尊重他人
- 不懂团队协作
- 推动变革的方式太激进

### 根回し的职业价值

掌握根回し = 证明你懂得：
- 尊重层级和经验
- 建立同盟
- 平稳推动变革

这是**Tech Lead 和架构师的核心能力**：

> 这些角色 50% 是技术，50% 是"社会工程"。

### 晋升的关键信号

当管理层考虑提拔你时，他们会问：

> 「この人は、チームをまとめて物事を進められるか？」
> （这个人能整合团队、推动事情进展吗？）

根回し就是证明"Yes"的最直接方式。
